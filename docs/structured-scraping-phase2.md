# 阶段 2：抓取结构化信息（开放时间、价格、活动）方案

在完成「1. 聚合链接/标题/时间」之后，若希望进一步自动获取**开放时间、价格、活动**等结构化信息，可参考本方案。难度和合规风险都高于只做链接聚合，建议小范围试点（例如只做 1～2 个稳定数据源）。

## 目标与范围

- **开放时间**：写入 `gyms.opening_hours`（或单独表，再与 gym 关联）。
- **价格**：写入 `gyms.price_note` / `gyms.price_single`，用于列表排序与问答推荐。
- **活动**：写入 `events` 表（标题、时间、链接等），与现有「活动 + 换线」展示一致。

以上均需**可追溯来源**（哪家馆、从哪个 URL/平台抓的、抓取时间），便于后续校对与合规。

## 数据源评估（按可行性）

| 类型           | 开放时间/价格 | 活动信息     | 说明 |
|----------------|---------------|--------------|------|
| 官网 / 活动页  | 若页面有固定结构可解析 | 同上，或 RSS | 需逐个站点写解析规则，结构一变就要改 |
| 大众点评/美团  | 有，但需解析页面或接口 | 较少         | 反爬与 ToS 限制多，不推荐优先 |
| 公众号         | 偶尔在推文里  | 有           | 无官方 RSS，需谨慎评估合规与风控 |
| 小红书/抖音    | 不统一        | 有           | 反爬强，V1 建议只做链接聚合 |

建议：**先选 1 个稳定、公开的数据源**（例如某一家馆的官网或提供 RSS 的页面），只针对该源写抓取逻辑，把结果写入 Supabase，再在前端展示。验证可行后再考虑扩展。

## 技术实现思路

1. **单独脚本或定时任务**（与 RSS 脚本类似）
   - 输入：数据源 URL、对应 `gym_id`、抓取类型（开放时间 / 价格 / 活动）。
   - 过程：请求页面或 API → 解析 HTML/JSON → 提取字段 → 校验格式。
   - 输出：upsert 到 `gyms`（开放时间、价格）或 `events`（活动），并可选写入「抓取日志表」（url、抓取时间、是否成功、原始片段），便于排查与合规。

2. **存储**
   - 开放时间 / 价格：直接更新 `gyms` 对应列，或新增 `gym_snapshots` 表按时间存历史，便于对比。
   - 活动：插入 `events`，注意去重（例如用活动标题+时间+来源 URL 生成唯一 id 或做 upsert）。

3. **频率与礼貌**
   - 抓取间隔建议不低于 1 小时，避免对目标站点造成压力；重要信息可辅以人工校对。

## 合规与风险

- 仅抓取**公开、非登录即可访问**的内容；遵守目标站点的 robots.txt 与使用条款。
- 不保存原文大段内容时，优先只存**结构化字段**（时间、价格、标题、链接），降低版权与合规风险。
- 若涉及公众号、小红书等平台，需单独评估其使用条款与反爬策略，必要时只做链接聚合（阶段 1）而不做正文或结构化抓取。

## 建议实施顺序

1. **先跑通阶段 1**：`social_posts` 表 + RSS 脚本 + 详情页「社交媒体动态」展示。
2. **选定一个试点源**：例如一家馆的官网「营业时间」页或活动列表页。
3. **为该源写专用脚本**：只解析开放时间/价格或活动，写入 Supabase，并在前端确认展示正确。
4. 若效果稳定，再考虑：增加抓取日志、扩展至其他馆或其它页面，并评估是否用 GitHub Actions / Cron 定时执行。
